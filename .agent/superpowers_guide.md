# 核心技能使用说明

这篇文档介绍了已集成到本项目的 **8 个核心技能（Skills）**。这些技能旨在为你进行复杂开发时，提供更系统、严谨、不易出错的高级工程师工作流。

我们配置了 **按需触发** 的模式，这意味着你可以通过简单的指令（例如：_"帮我设计一个新组件，请使用 brainstorming 技能"_ 或者 _"我们来修这个 Bug，启动 systematic-debugging"_）来唤醒它们。

> **技能分类**：
> - **工程流程**（#1-#5）：覆盖从设计到交付的完整开发周期
> - **视角与架构**（#6-#7）：跳出局部，理解和改善整体
> - **效率工具**（#8）：优化人机交互本身

---

## 1. `brainstorming` (设计与头脑风暴)

**【何时使用】**
在开始编写任何代码、创建新功能或重大重构之前。这是防止"想都不想就开始写代码"的最关键技能。

**【技能效果】**

- 它会**强制叫停**立即写代码的行为。
- 它会先研究项目现状，然后通过 **"每次只问一个选择题/澄清题"** 的方式，引导你把模糊的"点子"变成明确的需求。
- 接着，它会为你提供 **2-3 种不同的架构或实现方案** 供你挑选，并附带权衡分析（Trade-offs）和它的推荐。
- 最终，它会形成并向你展示一份结构化的设计说明，并在得到你的同意后，自动将设计内容导出。

**【唤醒示例】**

> _"我想做一个月报数据导出的功能，请使用 `brainstorming` 帮我构思一下。"_

## 2. `writing-plans` (计划拆解)

**【何时使用】**
当你已经有了一份设计草图或明确的需求说明，但在动手执行前需要一个周密的施工计划时。通常紧接着 `brainstorming` 之后使用。

**【技能效果】**

- 它会将你的总体设计，拆解为 **极其细致、每个耗时仅约 2-5 分钟的微小任务 (Bite-sized Tasks)**。
- 它的计划风格是为了让"毫无上下文的初级工程师"也能看懂，因此其中会精确到：**需要修改哪个文件的第几行、写什么样的代码、以及如何验证这步是成功的**。
- 一旦执行，它能极大避免中途思路混乱或遗漏分支逻辑。

**【唤醒示例】**

> _"我们上面讨论的方案很不错，按照那个设计调用 `writing-plans` 给我出一份执行步骤。"_

## 3. `test-driven-development` (严格型测试驱动开发)

**【何时使用】**
当你准备实现一个核心逻辑、复杂的工具函数，或非常害怕改坏现有代码的模块时。

**【技能效果】**

- 它会强制 Antigravity 采用 **红->绿->重构 (RED-GREEN-REFACTOR)** 循环。
- 它如果发现自己（或你）在没有写测试前就偷跑写了业务代码，**它会无情地把业务代码删掉/还原**，逼迫必须先写一个注定报错的 Test Case 并看着它 Fail（红）。
- 确认变红后，它才会编写"恰好能让测试通过的最小代码量"（绿），然后提交并重构代码以消除冗余。

**【唤醒示例】**

> _"帮我写一个按周筛选当月汇报数据的 util 函数，请严格按 `test-driven-development` 技能进行。"_

---

## 4. `systematic-debugging` (系统级除错)

**【何时使用】**
当你遇到诡异的 Bug、控制台报错或测试用例持续跑不通，同时你不希望它毫无逻辑地胡乱瞎改碰运气时。

**【技能效果】**

- 它会强制启动一套四阶段的"根因分析"排查法：**复现 -> 隔离 -> 假设检验 -> 深度防御**。
- 它会在代码里主动插入探针（如 `console.log` 面包屑或 Tracer），逐步缩小嫌疑范围，直到用数据证明找到了真正的 Root Cause。
- 修复后，它还会检查是否需要补充测试，并寻找相似联动的代码区域（以防其他地方存在同类隐患）。

**【唤醒示例】**

> _"这个表单提交时接口偶尔会多传一个字段报错，快帮我看一下，启动 `systematic-debugging`。"_

---

## 5. `requesting-code-review` (自动化代码审查)

**【何时使用】**
当一个功能模块刚写完，或者在重构任务的若干个步骤节点之间，你想要"质检"一下代码是否靠谱时。

**【技能效果】**

- 它相当于一位严厉的 Reviewer。它会将目前实现的代码与最初制定的 Plan 进行比对，检查"你到底有没有按照约定写"。
- 它还会执行标准的静态检查、边界遗漏检查、防御性编程检查等。
- 它会按严重程度报告发现的问题（如安全漏洞、不优雅的冗余代码等）。如果发现核心流程崩坏，它会判定为"阻塞性问题"并强烈建议先修好再推进下一步。

**【唤醒示例】**

> _"刚才这几个组件差不多搞定了，在你往下写或者提交之前，用 `requesting-code-review` 给我过一遍。"_

## 6. `zoom-out` (全局视角)

**【何时使用】**
当你对某段代码不太熟悉，或者陷入了局部细节看不清全貌时。适用于接手别人的代码、重构前评估影响范围、Debug 时跳出局部看全链路等场景。

**【技能效果】**

- 它会向上跳一层抽象，帮你绘制一张**模块地图**——列出所有相关的模块、调用者和被调用者。
- 它会梳理**数据流向**：数据从哪来、经过哪些模块、到哪去。
- 它会定位当前代码在整体架构中的**角色**（入口层 / 业务逻辑层 / 数据层 / 工具层）。
- 它会标注**上下文边界**和**耦合点**，告诉你"如果改了这里，哪里可能受影响"。
- 如果发现当前模块职责不清晰或过于庞大，会**主动指出**。

**【唤醒示例】**

> _"这个 `useFormSubmit` hook 我看不太懂它在整个系统里扮演什么角色，帮我 `zoom-out` 看看。"_
>
> _"重构这个模块之前，先用 `zoom-out` 帮我梳理一下影响范围。"_

## 7. `improve-codebase-architecture` (架构改善)

**【何时使用】**
当代码库越写越乱、模块之间耦合严重、想做大规模重构前的评估、或者只是想定期给代码"体检"时。**建议每隔几天运行一次**，防止技术债积累。

**【技能效果】**

- 它会有机地浏览代码库，寻找**架构摩擦点**——理解一个概念需要在多个小模块间跳来跳去、接口和实现一样复杂的浅层模块、紧耦合导致的接缝泄露等。
- 对每个可疑模块进行**删除测试**：想象删掉它，复杂性是消失了还是扩散到 N 个调用者中？
- 呈现一个编号的**加深机会清单**，每项包含：涉及文件、问题描述、改善方案、预期收益。
- 你选定某个候选方案后，它会进入**拷问式深度讨论**，帮你逐步推敲约束、依赖、模块形态和测试策略。
- 讨论中如果产生了新的领域概念，它会建议即时记录到 `CONTEXT.md`。

**【核心理念】**

> **深模块 vs 浅模块**：好的模块应该是"深"的——小接口背后藏着大量行为（高杠杆）。坏的模块是"浅"的——接口和实现一样复杂，调用者获得的杠杆很少。

**【唤醒示例】**

> _"最近代码写得有点乱，帮我跑一下 `improve-codebase-architecture`，看看有哪些地方可以优化。"_
>
> _"重构前先用 `improve-codebase-architecture` 做个架构评估。"_

## 8. `caveman` (原始人模式 🦴)

**【何时使用】**
当你觉得 AI 的回复太啰嗦、太多客套话、浪费太多 Token 时。开启后 Token 使用量降低约 75%，但技术精确度完全保持。

**【技能效果】**

- **砍掉一切废话**：冠词(a/an/the)、客套话("当然可以！我很乐意帮助你...")、犹豫语气、过渡句。
- **极致压缩**：用缩写(DB/auth/config/fn/impl)、箭头表因果(X -> Y)、句子片段代替完整句。
- **技术内容完全保留**：代码块不变、错误信息原样引用、技术术语精确。
- **安全例外**：遇到安全警告或不可逆操作时，自动临时恢复正常模式，确保你不会因为过度压缩而误操作。
- **持久生效**：一旦开启，后续每条回复都保持原始人模式，直到你主动关闭。

**【对比示例】**

| 正常模式 | 原始人模式 |
|---------|----------|
| 当然可以！你遇到的问题很可能是由于 auth 中间件中的 Token 过期检查逻辑使用了错误的比较运算符… | Bug 在 auth 中间件。Token 过期检查用 `<` 不是 `<=`。改: |

**【唤醒示例】**

> _"开启 `caveman` 模式"_
>
> _"原始人模式"_
>
> _"少说废话，简洁回答"_
